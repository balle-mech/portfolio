# TDD（テスト駆動開発）ルール

## 基本方針

本リポジトリでは TDD（Test-Driven Development）を採用する。
新機能・バグ修正のコードを書く前に、必ずテストを先に書く。

## Red → Green → Refactor サイクル

1. **Red**: 失敗するテストを書く
2. **Green**: テストが通る最小限の実装をする
3. **Refactor**: テストが通ったままコードを整理する

## 適用範囲

| 対象 | 方針 |
|---|---|
| カスタムフック（`hooks/`） | 必須。ロジックは hook に集約しテスト可能にする |
| ユーティリティ関数 | 必須 |
| React コンポーネント | 描画・インタラクションの主要パスを対象 |
| スタイルのみの変更 | テスト不要 |

## テストファイルの配置

テストファイルはテスト対象のソースファイルと**同じディレクトリ（コロケーション）**に置く。

- ファイル名: `{対象ファイル名}.test.ts` または `{対象ファイル名}.test.tsx`
- 例:
  - `src/hooks/useLaprasData.ts` → `src/hooks/useLaprasData.test.ts`
  - `src/components/HeroSection.tsx` → `src/components/HeroSection.test.tsx`
- `src/test/` のような集約ディレクトリは使用しない

**理由**: どのテストがどのファイルを検証しているかが一目でわかり、ファイル削除・移動時にテストの取り残しが起きにくい。

## 禁止事項

- テストなしで hooks/ や共通ロジックを追加しない
- `// @ts-ignore` や `any` でテストを回避しない
